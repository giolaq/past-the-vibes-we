Intro · the security model

The model is a contractor.
The harness is the foreman.

Model · read-only

  • Lists, reads, searches the guarded app
  • Returns a typed patch: {summary, files}
  • No shell. No write tool. Ever.

Harness · owns consequences

  • Writes files, runs checks, retries once
  • Enforces the cost cap, commits passing work
  • Keeps the receipts: evidence, hashes, reports
The model proposes; the harness decides. This boundary never moves today.

Lesson 1 · from prompt to loop

Start with one model call.
Then ask what it cannot prove.

$ tsx steps/01-single-agent/index.ts run fixtures/phases.json ✓ out/ANALYSIS.md written ✗ …is any of it true?
The file exists — that is all we know. The analysis could name components that don’t exist and call unportable code portable. Your job: write down three claims this script cannot prove.

Lesson 2 · verification and retry

A retry is only useful when it
carries new information

✗ Pattern "## Remote Control" not found in out/TV_PORT_PLAN.md → exact failure text goes into attempt 2 … ✓ plan repaired: remote control section added
“Try again” buys another guess. The exact failure narrows the problem and keeps the original requirement. One bounded retry — then stop.

Lesson 3 · phases, checkpoints, cost

Long runs fail.
Plan for the failure.

Three different memories

  • Phases limit the size of each change
  • Checkpoints remember run progress → --resume
  • Git commits preserve each verified code state

Watch for

  • checkpoint.json: {"nextPhase": 2}
  • Resume runs only build_test
  • Cost visible on every phase — and capped

Lesson 4 · tools, skills, executors

The skills are Amazon’s, not ours

analyze → amazon-devices-vega-best-practices plan → amazon-devices-vega-focus-management build_test → amazon-devices-vega-build-and-run
Installed by ADBT’s init-context, versioned by the vendor, consumed by the harness without owning a word of them. Swapping a phase’s expertise is a one-line change in phases.json. Claude CLI gets skill text in the prompt; Strands activates them through AgentSkills.

Under the hood · the agent runtime

Strands Agents SDK — the plumbing comes built in

The SDK supplies

  • One interface for Bedrock / OpenAI / OpenRouter
  • Zod-typed tools · schema-enforced output
  • Turn & token limits, cancellation, metrics, native MCP

It stays a library

  • Never asks to own writes, git, orchestration
  • The harness boundary survives the SDK
  • Our whole executor: ~100 lines
AWS’s open-source agent runtime, TypeScript and Python, pinned here at 1.10.0. The payoff of this split: control stays in your code, and SDK metrics plus harness evidence give you observability over every run.

ADBT — vendor docs, not guessed APIs

One pinned npm package

Lesson 7 · TV as the stress test

A screenshot shows a moment.
Focus is temporal.

launch → featured action has focus down → focus enters the first rail left/right → stops at boundaries select → details opens for that card back → the same card regains focus
The last line is the one screenshots always miss. It’s verified by an executable check, not an eyeball: tv-focus-result.json → "passed": true.

Lesson 8 · the Vega device lifecycle

Eight gates, and every claim
labeled live or replay

sdk → device → build → install → launch → logs → capture → pull evidenceMode: live this app ran on an attached device evidenceMode: replay the control flow works — not device proof
A successful process exit is not evidence when no device is attached. The harness records the exact command, exit code, artifact, and mode for every gate — keep the two claims separate.
